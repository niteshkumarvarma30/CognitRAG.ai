# Notion Integration - End-to-End Architecture

This document outlines the complete architecture and data flow for the Notion Integration built for CognitRAG.ai. The integration allows users to securely connect their Notion workspaces via OAuth 2.0 and automatically ingest pages into the hybrid graph RAG memory system.

## 1. OAuth 2.0 Authentication Flow

The authentication process uses Notion's public API to grant CognitRAG secure, scoped access to the user's workspace.

1. **Frontend Initiation (`frontend/app/app/(dashboard)/integrations/page.js`)**:
   - The user clicks the **"Connect Notion"** button on the integrations dashboard.
   - The button directs the user to the backend authorization endpoint: `/api/v1/integrations/notion/auth?tenant_id={clerk_user_id}`.

2. **Backend Authorization (`src/api/integration_routes.py`)**:
   - The `/auth` endpoint builds the Notion OAuth URL using the `NOTION_CLIENT_ID` and redirect URI (`http://localhost:8000/api/v1/integrations/notion/callback`).
   - The user's `tenant_id` is passed as the `state` parameter to securely track the request across the redirect.
   - The user is redirected to Notion's official consent screen, where they select which pages CognitRAG can access.

3. **Token Exchange & Storage (`/callback`)**:
   - After granting access, Notion redirects back to the `/callback` endpoint with a temporary `code`.
   - The backend makes a secure POST request to `api.notion.com/v1/oauth/token` using the `NOTION_CLIENT_SECRET` (Basic Auth) to exchange the code for a permanent `access_token`.
   - The `access_token`, `workspace_name`, and Notion `workspace_id` are saved to the `tenant_integrations` table in Supabase.
   - The user is finally redirected back to the frontend dashboard (`http://localhost:5173/integrations?status=success`).

## 2. Database Schema (Supabase)

To securely map third-party integrations to specific Clerk users (tenants), a new table was created.

### Table: `public.tenant_integrations`
- `id` (uuid, primary key): Unique identifier for the record.
- `tenant_id` (text): The Clerk User ID (e.g., `user_2b...`). Links the integration to the exact user workspace.
- `integration_type` (text): Set to `'notion'` (allows for future expansion like Google Drive).
- `access_token` (text): The encrypted OAuth token provided by Notion.
- `workspace_name` (text): The human-readable name of the Notion workspace.
- `workspace_id` (text): The internal Notion workspace ID.
- `created_at` / `updated_at` (timestamptz)

**Security (Row Level Security):**
- RLS is fully enabled on this table.
- A policy `Tenants can manage their own integrations` restricts users to only SELECT, INSERT, UPDATE, or DELETE rows where the `tenant_id` matches `auth.jwt() ->> 'tenant_id'` mapped from their active Clerk session.

## 3. Data Ingestion & Crawler (`src/integrations/notion_crawler.py`)

Once the integration is authorized, the crawler script handles pulling the documents down from Notion and piping them into the RAG pipeline.

1. **Authentication Check**: 
   The script looks up the `tenant_id` in `tenant_integrations` and retrieves the secure `access_token`.
   
2. **Page Discovery**:
   - The crawler hits the Notion Search API (`POST https://api.notion.com/v1/search`) to discover every page the user explicitly checked during the consent screen.
   - It iterates through the paginated results, extracting the internal `page_id` and the plaintext `title`.

3. **Document Registration**:
   - Before downloading content, it registers a new document inside the `documents` table in Supabase.
   - `filename` is set to `Notion: {title}` and `status` is set to `pending`.
   - A unique `document_id` is generated by Supabase to track the ingestion state.

4. **Block Recursion & Extraction**:
   - The crawler queries the Notion Blocks API (`GET https://api.notion.com/v1/blocks/{page_id}/children`).
   - It iterates through the blocks (paragraphs, headers, lists) and extracts the `plain_text` from the `rich_text` arrays to construct a clean Markdown-style document.

5. **Vector & Graph Processing**:
   - The plaintext page is fed into `chunk_text()` from `src.ingestion.parser`.
   - The chunk array is then piped into `run_ingestion_pipeline(tenant_id, document_id, chunks)` (`src.ingestion.pipeline`), where it is parsed by LLMs, embedded via Jina AI, and relationships are mapped into the Neo4j Knowledge Graph.

6. **Completion State**:
   - Upon successful ingestion, the document status is updated to `completed` in Supabase. If any step fails, it is marked as `failed`.

## 4. Frontend Resilience Updates

During development, we also discovered and fixed critical frontend session state bugs related to Clerk:
- **Middleware Protections (`middleware.js`)**: Configured Clerk to asynchronously await token validation and correctly use `NextResponse.redirect` alongside URL query flags `?sign_out=true` to handle expired JWT tokens robustly.
- **Client Cache Synchronization**: Updated the React landing page (`page.js`) to intercept the `sign_out=true` URL flag, forcefully clearing the stale frontend state (`signOut()` via `useAuth`) to prevent "ghost sessions" where the UI thinks the user is logged in, but the backend rejects the requests.
- **Clerk Modal Fix**: Swapped out broken local `/sign-in` routing logic for Clerk's native, bulletproof `<SignInButton mode="modal">`.
